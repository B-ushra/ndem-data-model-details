📋 NDEM Data Model Overview
The NDEM data model is built on NIEM principles, using HAS-A composition relationships to create a semantic data structure for heterogeneous multi-UAV systems. The model follows strict hierarchical composition where complex objects contain and manage their component objects.
NDEMMessage (Root)
├─ messageID [1]: EString
├─ timestamp [1]: EDate
├─ HAS-A Header [1]
├─ HAS-A UavIdentificationData [1]
├─ HAS-A UavTelemetryData [0..1]
├─ HAS-A UavSensorData [0..1]
├─ HAS-A UavCommandAndControlData [0..1]
└─ HAS-A UavMissionData [0..1]

Core Concepts

🎯 Composition Pattern (HAS-A)
Objects within NDEMMessage are composed rather than referenced. Each parent object owns and manages its child objects. This ensures:
  • Data Integrity: Complete object graphs in single messages
  • Semantic Completeness: All related data travels together
  • Protocol Independence: Transformations handle entire compositions
  • NIEM Compliance: Follows national information exchange model principles
🏗️ Cardinality Rules
[1]: Required - must be present in every message
[0..1]: Optional - may be present depending on message type
Nested [0.1]: Child objects with 0 or 1 instances (scalar composition)
📊 Typical Message Data Flow
Fixed-Wing UAV generates observation → Creates UavObservation object
UavObservation composed into UavSensorData
UavSensorData composed into NDEMMessage with complete context
GCS receives complete object graph → Transforms for other systems
DJI SDK receives mapped telemetry → Executes commands

Header - Message Routing
Header
<<dataType>> Message routing and identification
The Header class encapsulates message routing information, defining the source (sender) and destination (receiver) of NDEM messages in the multi-UAV coordination network.
Attributes
senderID
String - Source system identifier
Required
receiverID
String - Destination system identifier
Required
Naming Convention for IDs
[PlatformType]-[Protocol]-[Instance]

Examples:
• FixedWing-MAVLink-001 (reconnaissance drone)
• DJI-Quadcopter-Thermal-02 (thermal imaging)
• Crazyflie-Indoor-03 (indoor exploration)
• GCS-CommandCenter (ground station)
• IncidentCommand-SAR (incident command system)
UavIdentificationData - Platform Metadata
UavIdentificationData
<<dataType>> UAV platform and firmware identification
Encapsulates identifying information about the UAV platform, including manufacturer, model, platform type, and firmware version. This is essential for protocol selection and semantic mapping.

String [1] - Platform model (Iris+, Phantom 4, Crazyflie 2.1)
Required
uavType
String [1] - Aircraft type category
Required
firmwareVersion
String [0..1] - Firmware/software version
Optional
Role in NDEM: The identification data is used by the semantic mapping layer to:
  • Determine appropriate protocol handlers (MAVLink, DJI SDK, CRTP)
  • Select correct transformation mappings
  • Validate platform capabilities against mission requirements
  • Route to appropriate subsystems

Cardinality Semantics

Understanding Cardinality Markers
[1]
Required - Must be present in every message. Parent object is invalid without this child.
[0..1]
Optional - May or may not be present depending on message context. If present, exactly one instance.
[0.1]
Scalar Optional - At most one instance of this component within parent.

Core NIEM-Inspired Types

Entity & Location Core Types
<<core>> Entity (ncEntity): Represents physical or logical device/person. Used by UAVs and organizational entities.

<<core>> Location (ncLocationType): Represents geographic location with coordinates. Used by TargetLocation and UavPosition.
